1) Heredoc 使用不一致 — 存在注入/本地展开风险

问题：在主流程中使用了 _ssh::pipe << SHEOF（未使用单引号），示例片段：_ssh::pipe << SHEOF 并在 heredoc 中直接插入了 LINK_ID 等变量（这些变量在本地被展开然后发送到中转机）。脚本头部的 FIX-2 明确建议使用 << 'SHEOF' + bash -s -- args 模式以防止本地展开；但实际主流程仍然使用不带引号的 heredoc。请见：远端执行片段示例。

duijie_wg_v4.0

风险与说明：

如果某些变量（例如从落地机读取的公钥/域名）包含意想不到的字符（比如反引号、命令替换、特殊 shell 扩展等），未加引号的 heredoc 会导致本地先展开、再发往远端，从而产生注入或意外解释。

虽然脚本对大多数字段来源是可信（例如来自自己机器或 luodi.sh），良好实践仍应消除理论注入面。

建议修复

将所有通过 _ssh::pipe 发送多行远端脚本的地方改为使用 _ssh::pipe_args + 单引号 heredoc。示意：

# 本地传参（示例）
_ssh::pipe_args "$LINK_ID" "$RELAY_ASSIGNED_PORT" "$LUODI_WG_ASSIGNED_IP" << 'REMOTE_EOF'
LINK_ID="$1"
RELAY_PORT="$2"
WG_IP="$3"
# … 剩余脚本使用 $LINK_ID/$RELAY_PORT/$WG_IP，不会被本地 shell 展开
REMOTE_EOF

如果因为便利仍要用 _ssh::pipe << EOF，则确保所有变量都经过严格的 shell-escaping（麻烦且容易出错），所以优先改为 pipe_args 为最佳。

引用脚本中已经实现的安全函数 _ssh::pipe_args()：

4) 回滚与原子性策略欠缺（部分成功 / 部分失败时需回滚）

问题：脚本步骤包括：分配 WG IP → 在中转机添加 WG peer → 在落地机配置本地 wg0 → 分配外部端口 → 写 iptables DNAT。若任一步失败（例如 iptables 写入失败或中转机重启），脚本里没有完整的回滚步骤来撤销已经添加的 peer、还原 peer_map、清除部分 iptables 规则等（我在主流程与各模块没有看到一个集中式的 on_error 回滚逻辑）。
